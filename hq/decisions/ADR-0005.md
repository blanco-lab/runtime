# ADR-0005: Runtime Services / Service Manager (pieza futura)

- Estado: PROPUESTO (no implementar todavía)
- Fecha: 2026-07-18
- Origen: revisión AR-002 (Atlas)

## Contexto

La primera Capability real (music_player, AR-002) reveló una dependencia
no evidente hasta ahora: **Runtime depende del estado del sistema**.

El backend spotify_player solo funciona si existe un dispositivo de
Spotify Connect activo. Lo resolvimos con un servicio systemd de usuario
(`spotify-player-daemon.service`), pero eso fue una solución puntual,
externa al núcleo de Runtime.

El descubrimiento importante de esta iteración no fue Spotify, sino que
Runtime necesitará **garantizar que los servicios de los que dependen las
capabilities estén preparados antes de ejecutarlas**.

## Decisión (propuesta)

Anotar como pieza futura del núcleo:

    Runtime Services / Service Manager

Responsabilidad: asegurar que todos los servicios necesarios estén
listos antes de ejecutar una capacidad. Ejemplos de servicios:

    spotify_player, ollama, postgres, redis, telegram, suitecrm, ...

Comportamiento futuro esperado (a diseñar):

1. Cada Backend/Capability declara sus dependencias de servicio.
2. Antes de ejecutar, el Service Manager comprueba/arranca los servicios
   requeridos (systemd u otro supervisor).
3. Si un servicio no puede prepararse, la ejecución falla de forma clara
   (sin 404 crípticos como el de esta iteración).

## Estado

NO implementar ahora. Queda registrado como componente del núcleo a
diseñar cuando el número de dependencias de servicio lo justifique.
La solución actual (servicio systemd de usuario para spotify_player)
es válida como puente hasta entonces.
