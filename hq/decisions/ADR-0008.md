# ADR-0008: Reutilizar auth existente antes de pedir una nueva

- Estado: ACEPTADO
- Fecha: 2026-07-19
- Decisores: Blanco (Director), Atlas (Chief Architect)
- Origen: reversión de la lectura estricta de ADR-0007 (opción b) a raíz
  del problema de UX de los podcasts.

## Contexto

ADR-0007 eligió la opción (b): cada usuario registra su propia Spotify
App y se autentica con su propio client_id. Arquitectónicamente limpia,
pero la experiencia demostró que es inaceptable: exigir que un usuario
sea desarrollador de Spotify solo para oír un podcast es una barrera
demasiado alta. Para música el usuario no hizo nada (spotify_player ya
le autenticó); para podcast exigirle registrar una app rompe la
simetría.

Blanco planteó el principio: "lo razonable es que el usuario autentique
SU CUENTA (login una vez); no que registre una APP DE DESARROLLADOR."

## Decisión (revisión de ADR-0007)

Se adopta una VARIANTE de la Opción 3 de Hermes, entendida como
POLÍTICA GENERAL de Runtime (no como parche de Spotify):

    Runtime siempre aprovechará una autenticación existente antes de
    solicitar una nueva.

Concretamente:

1. Si ya existe una sesión válida (spotify_player, navegador, futuro
   Horizon, etc.), Runtime la reutiliza.
2. Solo cuando NO exista ninguna identidad reutilizable, Runtime inicia
   su propio proceso de autenticación (AuthManager / PKCE).

Esto convierte la autenticación en política general, no en caso especial
de Spotify.

## Implementación

SpotifyBackend dispondrá de DOS proveedores de credenciales, detrás de
una interfaz única. El backend SOLO pide "dame una credencial válida";
no le importa el origen (pertenece a la infra de auth):

- `ExistingCredentialProvider`: reutiliza una credencial existente (HOY:
  el token de spotify_player en ~/.cache/spotify-player/). Es el PRIMER
  proveedor intentado.
- `RuntimeCredentialProvider`: usa la infra AuthManager (PKCE) cuando no
  haya credencial previa. Es el SEGUNDO intento.

La lógica de resolución (intentar existente → si falla, OAuth) vive en
la INFRAESTRUCTURA DE AUTENTICACIÓN (`src/core/auth/`), NO dentro del
backend Spotify.

## Restricciones preservadas

- A1 (Auth ≠ Tokens): el almacén interno sigue siendo genérico
  (Credential), no acoplado a OAuth.
- A2 (ProviderAdapter no guarda): los adapters devuelven credenciales;
  quien persiste es siempre AuthManager. ExistingCredentialProvider
  tampoco guarda: solo LEE una sesión ya existente.
- A3 (device_id desacoplado): SpotifyWebApiTransport recibe device_id
  inyectado, sin saber de dónde procede.

## Lo que NO se elimina

No se revierte la infra de ADR-0007: AuthManager, ProviderAdapter y
PKCE SE MANTIENEN. Solo se AÑADE un proveedor de credenciales
reutilizables por delante del flujo OAuth.

## Notas de evolución

- spotify_player NO es requisito permanente: es simplemente el primer
  ExistingCredentialProvider. En el futuro podrían existir otros (p.ej.
  Horizon como proveedor de credenciales). Por eso esta lógica vive en
  la infra de auth, no en el backend Spotify.

## Consecuencias

- El usuario (Blanco) no hace nada nuevo para oír un podcast: Runtime
  reusa el token de spotify_player.
- Un usuario SIN spotify_player sí deberá autenticarse una vez (login
  OAuth, sin ser desarrollador).
- Parte B (reproducción de podcasts) se implementa sobre esta política.
