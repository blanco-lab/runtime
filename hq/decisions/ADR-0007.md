# ADR-0007: Auth por proveedor, propiedad del usuario

- Estado: ACEPTADO
- Fecha: 2026-07-19
- Decisores: Blanco (Director), Atlas (Chief Architect)
- Origen: refinamiento de Opción B (podcasts), respuesta de Atlas

## Contexto

Para reproducir podcasts (y, en general, para cualquier backend que hable
con un servicio externo) Runtime necesita autenticación del usuario contra
ese servicio. La primera opción propuesta reutilizaba el token de
spotify_player de Blanco, acoplando Runtime a su instalación y a su cuenta.

Blanco planteó el principio correcto: Runtime se construye para que OTRAS
PERSONAS lo usen; no puede venir preconfigurado con sus datos.

## Decisión

1. **Opción (b) confirmada.** Cada usuario es propietario de su propia
   autenticación. Runtime NO distribuye un Spotify App ni depende de
   credenciales de Blanco. Cada instalación proporciona su propio
   client_id/client_secret y autentica mediante un flujo OAuth propio de
   Runtime. La autenticación pertenece al usuario, nunca al desarrollador.

2. **Mecanismo general `runtime auth <provider>`.** Runtime ofrece un
   comando de autenticación por proveedor:
       runtime auth spotify
       runtime auth google      (futuro)
       runtime auth github      (futuro)
       runtime auth telegram    (futuro)
   Cada backend implementa su autenticación; Runtime solo proporciona el
   mecanismo común para almacenar y renovar credenciales.

3. **Almacenamiento por usuario.** Token en:
       ~/.config/runtime/auth/<provider>.json
   (p.ej. ~/.config/runtime/auth/spotify.json).

4. **No cerrar la puerta, sin generalizar hoy.** Se implementa Spotify
   diseñando sin acoplarse a "solo Spotify"; NO se implementa toda la
   generalización ahora. La estructura debe evolucionar naturalmente hacia
   múltiples proveedores.

## Consecuencias

- El backend (SpotifyBackend) implementa su estrategia de auth internamente.
- La Capability music_player sigue CONGELADA; solo evoluciona el backend.
- Se preserva ADR-0004 (Runtime no acoplado a una herramienta concreta).
- El flujo OAuth es trabajo del backend, no de la Capability (restricción
  de Atlas: nada de REST en la Capability).
