# ADR-0003 — Mejoras arquitectónicas post-MVP (revisión de Atlas)

Estado: APPROVED (revisión de Architect)
Fecha: 2026-07-18
Responsable: Atlas (Chief Architect)
Registrado por: Hermes (Platform Engineer)

## Contexto

Tras la construcción del primer pipeline ejecutable (commit 9f1d24a:
Intent → Action → Safety → Executor → Reporter, con Executor mock),
Atlas revisa el código y lo aprueba sin cambios. Propone dejar registradas
tres decisiones arquitectónicas para no tener que reestructurar más adelante.

## Decisiones

### 1. Safety tendrá tres estados: ALLOW / CONFIRM / REJECT

Hoy el mock solo distingue ALLOW/REJECT. Se añadirá CONFIRM para acciones
sensibles (ej. "Apaga el ordenador") que requieran confirmación humana en
lugar de rechazarse. La lista negra de capacidades prohibidas (defensa en
profundidad) se mantiene para lo verdaderamente inaceptable.

### 2. Intent incorporará los campos `confidence` y `entities` desde el inicio

Aunque por ahora sean valores mock, la estructura de Intent debe incluir
`confidence` (float) y `entities` (dict) para evitar romper la interfaz
cuando se conecte un proveedor de IA real.

### 3. Planner será una pieza futura del pipeline

Para acciones complejas (ej. "Buenos días Runtime"), el pipeline incorporará
en el futuro un Planner entre Intent/Action y el resto. NO se implementa hoy.
Se deja constancia de que existirá.

## Consecuencias

- El Executor evolucionará hacia un "Capability Executor" capaz de resolver
  automáticamente quién ejecuta cada acción (Spotify, Telegram, SuiteCRM,
  script, IA...). Proveedor-agnóstico. No se toca ahora.
- El MVP actual queda como base funcional; estas decisiones guían la
  siguiente fase sin bloquearla.
